This is an x86-64 bomb for self-study students. 

本目录包含用于构建和运行 CS:APP Bomb Lab（二进制炸弹实验）所需的文件。该实验旨在帮助学生掌握机器级程序的原理，以及通用的调试器使用和逆向工程技能。

## 1. 概述 (Overview)

### 1.1. 二进制炸弹 (Binary Bombs)

“二进制炸弹”是一个运行在 Linux 上的可执行 C 程序，由六个“关卡（阶段）”组成。每个关卡都要求学生在标准输入（stdin）中输入一个特定的字符串。如果学生输入了符合预期的字符串，该关卡就会被“拆除（defused）”。否则，炸弹就会“爆炸（explodes）”并打印出 `"BOOM!!!"`。学生的目标是尽可能多地拆除这些关卡。

### 1.2. 破解二进制炸弹 (Solving Binary Bombs)

为了拆除炸弹，学生必须使用调试器（通常是 GDB 或 DDD）对二进制文件进行反汇编，并在每个关卡的机器码中进行单步调试。核心思路是理解每条汇编语句的作用，然后根据这些知识推导出能够解密该关卡的字符串。

学生成功拆除关卡可以获得分数，而每次炸弹爆炸都会被扣分（扣分额度可由讲师配置，通常每次扣 0.5 分）。因此，学生们很快就会学乖，懂得在进入每个关卡以及触发炸弹爆炸的函数之前设置断点。这是一个极佳的教学实践，能强迫他们学会使用调试器。

### 1.3. 自动评分服务 (Autograding Service)

我们创建了一个独立的用户级自动评分服务，为您全权处理炸弹实验的所有事务：学生从服务器下载各自的炸弹。在学生破解炸弹的过程中，每次爆炸和拆除的数据都会实时流式传回服务器，并在 Web 端的“实时记分板（Scoreboard）”上显示每份炸弹的当前结果。本实验无需学生显式提交作业，完全实现自动评分。

该自动评分服务由运行在 `./bomblab` 主目录下的四个用户级程序组成：

- **请求服务器 (Request Server - `bomblab-requestd.pl`)：** 学生通过浏览器访问这个简单的 HTTP 服务器（即“请求服务器”）来下载各自的炸弹并查看记分板。请求服务器会构建炸弹，将其打包成 tar 文件，然后将生成的 tar 文件上传回浏览器，供学生保存到磁盘并解压。请求服务器还会为讲师保留一份该炸弹及其答案的副本。
- **结果服务器 (Result Server - `bomblab-resultd.pl`)：** 每当学生成功拆除炸弹关卡或引发爆炸时，炸弹就会向 HTTP “结果服务器”发送一条简短的 HTTP 消息（称为“自动结果字符串/autoresult string”），该服务器只需将该字符串追加到“记分板日志文件”中。
- **报告守护进程 (Report Daemon - `bomblab-reportd.pl`)：** 报告守护进程会定期扫描记分板日志文件。它会找出每个学生针对每个关卡提交的最新拆除字符串，并通过在学生炸弹的本地副本上运行这些字符串来进行验证。随后，它会更新 HTML 记分板，该记分板按累计总分对炸弹进行排名，并汇总每份炸弹当前的爆炸和拆除次数。
- **主守护进程 (Main daemon - `bomblab.pl`)：** 主守护进程负责启动并监视（nannies）请求服务器、结果服务器和报告守护进程，确保在任何时间点，这些进程（以及它自身）中都有且仅有一个实例在运行。如果其中某个进程因某种原因崩溃，主守护进程会检测到并自动将其重启。主守护进程是您实际唯一需要手动运行的程序。

## 3. 炸弹术语 (Bomb Terminology)

- **LabID：** 每次开设该实验（例如不同学期或班级）都由讲师指定一个唯一的名称来标识，例如 `"f12"`（12年秋季）或 `"s13"`（13年春季）。如果炸弹提交的 LabID 与当前活跃的 LabID 不一致，其记录将被忽略。LabID 中不得包含空格。
- **BombID：** 在特定实验实例中，每个炸弹都有一个唯一的非负整数标识，称为 `"bombID"`。
- **通知型炸弹 (Notifying Bomb)：** 炸弹在编译时如果启用了 `NOTIFY` 选项，每当学生引发爆炸或拆除关卡时，炸弹都会自动发送一条 HTTP 消息给服务器。这类炸弹被称为“通知型炸弹”。
- **静默型炸弹 (Quiet Bomb)：** 如果使用 `NONOTIFY` 选项进行编译，则炸弹在爆炸或被拆除时不会向服务器发送任何消息。

为了方便，我们还需要区分**定制炸弹**和**通用炸弹**：

- **定制炸弹 (Custom Bomb)：** 其 BombID 大于 0，与特定学生绑定，可以是通知型也可以是静默型。**定制通知型炸弹被限制在讲师指定的特定 Linux 主机集上运行**。相比之下，定制静默型炸弹可以在任何 Linux 主机上运行。
- **通用炸弹 (Generic Bomb)：** 其 BombID 等于 0，不与任何特定学生绑定，属于静默型，因此可以在任何主机上运行。

## 4. 开设炸弹实验 (Offering the Bomb Lab)

炸弹实验主要有两种开展形式：

1. **在线版 (Online)：** 讲师使用自动评分服务，根据学生的需求在网页端分发定制的通知型炸弹，并在实时记分板上自动追踪他们的进度。
2. **离线版 (Offline)：** 讲师手动构建、分发并批改学生的炸弹，不使用任何自动评分服务。

虽然这两种版本都能给学生带来丰富的体验，但我们**强烈推荐在线版本**。对学生来说，这显然最具吸引力也最有趣；对讲师来说，分发和评分也最省心。然而，这需要您在整个实验期间让自动评分服务保持不间断运行。我们把运行这个服务做得非常简单，但如果有些讲师不希望维护后台常驻服务，可以选择离线版本。

以下是开展这两种版本实验的具体指南：

### 4.1. 创建炸弹实验目录

确定你要创建炸弹实验目录（`./bomblab`）的通用 Linux 机器（`$SERVER_NAME`）。如果开设在线版本，该机器也将运行自动评分服务。你只需要拥有这台机器上的一个普通用户账户即可，**不需要 root 权限**。

每次新开设炸弹实验时，都需要在 `$SERVER_NAME` 上从一个干净全新的 `./bomblab` 目录开始。例如：

Bash

```
linux> tar xvf bomblab.tar
linux> cd bomblab   
linux> make cleanallfiles
```

### 4.2 配置炸弹实验

通过编辑以下文件来配置炸弹实验：

- **`./Bomblab.pm`** - 这是主配置文件。你只需要修改或检查该文件第一部分（Section 1）中的少数几个变量。每个变量前面都有详细的注释说明。如果你开设的是离线版本，可以忽略其中大部分设置。

如果你开设的是在线版本，还需要编辑以下文件：

- **`./src/config.h`** - **该文件列出了允许运行通知型炸弹的主机域名。请务必正确更新此文件**，否则你和你的学生将无法运行下载的炸弹（运行会报错提示主机受限）。

### 4.3. 更新实验说明书

更新配置文件后，根据你的具体教学环境修改 `./writeup/bomblab.tex` 中的 LaTeX 实验说明书。然后在 `./writeup` 目录中输入以下命令：

Bash

```
unix> make clean
unix> make 
```

这将在该目录下生成 PostScript (ps) 和 PDF 版本的实验手册。

### 4.4. 运行在线炸弹实验 (Online)

#### 4.4.1. 精简版流程

在 `./bomblab` 目录下：

1. **彻底重置实验数据（从头开始）：**

   Bash

   ```
   linux> make cleanallfiles
   ```

2. **启动自动评分服务：**

   Bash

   ```
   linux> make start
   ```

3. **停止自动评分服务：**

   Bash

   ```
   linux> make stop
   ```

你可以根据需要随时启动和停止自动评分服务，**这不会丢失任何已有的学生解题数据**。遇到任何不确定的状态时，执行 `make stop; make start` 可以让所有服务恢复到稳定状态。

> ⚠️ **注意：** `make cleanallfiles` 重置操作会删除所有已生成的炸弹、状态日志和记分板历史记录。请仅在调试期间，或者第一次对学生开放实验时才执行此操作。

- 学生申请炸弹的浏览器地址：`http://$SERVER_NAME:$REQUESTD_PORT/`
- 学生查看记分板的浏览器地址：`http://$SERVER_NAME:$REQUESTD_PORT/scoreboard`

#### 4.4.2. 详细版说明

**（1）重置炸弹实验：**

`make stop` 可确保没有服务器后台在运行。`make cleanallfiles` 会清除当前实验实例的所有数据（如状态日志、请求服务器为学生创建的所有炸弹以及记分板日志）。当您准备好让实验对学生“正式上线”时，请执行此操作。

在筹备实验期间，重置功能也非常有用。在正式上线前，你可以为自己申请几个炸弹、运行它们、拆除几个关卡、故意引爆几次，以此确保记分板能正确显示结果。如果出现问题（例如你忘记在 `src/config.h` 中更新允许运行炸弹的机器列表），你可以修正配置，重置实验，然后重新测试。

> 🛑 **严正警告：** 如果在实验上线后（学生已经开始做题）你执行了重置，你将丢失学生炸弹的所有记录及其对应答案，你将无法验证学生的提交，并且你的学生必须重新获取新的炸弹并从头开始。

**（2）启动炸弹实验：**

`make start` 会运行 `bomblab.pl`（主守护进程）。它会拉起并持续监视其他服务程序，每隔几秒检查一次它们的状态，并在必要时重新启动它们。

**（3）停止炸弹实验：**

`make stop` 会杀死所有正在运行的后台服务器。这不会丢失任何学生数据。

- **请求服务器 (Request Server)：** 是一个简单的专用 HTTP 服务器。

  学生申请炸弹分为两步：首先，学生在浏览器中访问 `http://$SERVER_NAME:$REQUESTD_PORT/`（例如 `[http://foo.cs.cmu.edu:15213/](http://foo.cs.cmu.edu:15213/)`）。请求服务器会返回一个 HTML 表单。接下来，学生填写自己的用户名和电子邮箱并提交。服务器解析表单，构建并打包一个 `bombID=n` 的通知型定制炸弹，并将该 tar 文件交付给浏览器下载。学生将其解压后，会得到一个目录（`./bomb<n>`），包含以下四个文件：

  - `bomb*`：通知型定制炸弹可执行文件
  - `bomb.c`：炸弹主主框架代码（主要显示 main 函数，核心关卡代码已隐去）
  - `ID`：标识与此炸弹关联的学生信息
  - `README`：列出炸弹编号、学生和电子邮箱地址

  同时，服务器也会在后台的 `bomblab/bombs/bomb<n>` 目录为讲师生成一套完整的备份，包括该学生的专属解密答案 `solution.txt` 以及用于自动评分的静默版炸弹 `bomb-quiet*`。

- **结果服务器 (Result Server)：** 每当学生拆除一个关卡或引爆炸弹时，炸弹会自动向其发送一条 autoresult 消息，包含 BombID、关卡号以及事件类型（如果拆除成功，还会附带学生输入的正确解密字符串）。服务器会将其追加到记分板日志中。

- **报告守护进程 (Report Daemon)：** 定期扫描日志，统计每个炸弹的爆炸次数、最高通关数，使用静默版炸弹对学生的答案进行本地验证，并在 `scores.txt` 中更新每个学生的分数。

#### 4.4.3. 在线炸弹实验的评分

在线炸弹实验完全是**自动评分**的。在任何时间点，制表符分隔的文本文件（`./bomblab/scores.txt`）都包含每个学生的最新分数。该文件由报告守护进程在每次生成新记分板时同步自动更新，讲师直接导出该文件即可作为最终成绩。

#### 4.4.4. 在线版本的附加注意事项

- 由于请求服务器和报告守护进程都需要在本地执行炸弹以进行验证，你**必须**在 `bomblab/src/config.h` 文件的合法机器列表中包含服务器自身的 `$SERVER_NAME`。

- 所有后台服务都是无状态（stateless）的。如果 Linux 服务器不小心崩溃或重启了，或者你修改了守护进程的代码，直接执行 `make stop` 确保清理干净，然后执行 `make start` 即可完美恢复，不会有任何副作用。

- 系统日志会记录在 `bomblab/log-status.txt` 中。由于它们运行在后台且不输出标准输出，你可以很方便地将 `make start` 配置为系统的开机自启脚本（如通过 `initrc`）。

- 讲师上线前的完整测试推荐流程：

  Bash

  ```
  linux> make cleanallfiles   
  linux> make start
  # 浏览器访问下载测试炸弹 -> 解压 -> 在允许的机器上运行、故意引爆和拆除几次 -> 检查记分板页面是否刷新
  # 确认一切正常后，重置为干净的生产环境向学生开放：
  linux> make stop
  linux> make cleanallfiles
  linux> make start
  # 此时正式上线，此后切勿再输入 make cleanallfiles。
  ```

### 4.5. 运行离线炸弹实验 (Offline)

在此版本中，你不需要运行任何后台自动评分服务。你需要手动构建一批“静默型炸弹”并分发给学生。学生独立完成拆除后，向你提交他们的答案文件，由你或助教进行手动批改评分。

你可以使用 `makebomb.pl` 脚本手动构建炸弹及其对应的标准答案。输入 `./makebomb.pl -h` 可查看参数详情。

#### 方案 1：单一通用炸弹（最省事，但不推荐）

让全班所有学生尝试拆除同一个完全相同的通用炸弹：

Bash

```
linux> ./makebomb.pl -s ./src -b ./bombs
```

这将在 `./bombs/bomb0` 中创建一个通用炸弹。你只需要把其中的 `bomb`（可执行文件）和 `bomb.c` 分发给学生即可。

学生写完后提交他们的答案文本文件（例如 `student_solution.txt`），你通过重定向输入来验证是否通关：

Bash

```
linux> cd bombs/bomb0
linux> ./bomb < student_solution.txt
```

- **评语：** 这种方案对老师来说最简单，但**强烈不推荐**，因为所有人的题目和答案完全一样，学生之间极其容易发生抄袭和作弊。

#### 方案 2：为每个学生定制专属离线炸弹（推荐）

利用脚本为每个学生生成一个包含唯一关卡密钥的专属静默炸弹：

Bash

```
linux> ./makebomb.pl -i <n> -s ./src -b ./bombs -l bomblab -u <学生邮箱> -v <学生用户名>
```

这将在 `./bombs/bomb<n>` 中为该学生生成专属题目。你需要向学生分发生成的 `bomb`、`bomb.c`、`ID` 和 `README` 四个文件。

批改时，进入该学生对应的后台备份目录，用他的专属炸弹去跑他提交的答案文件即可：

Bash

```
linux> cd ./bombs/bomb<n>
linux> ./bomb < student_n_solution.txt
```

各个不同关卡变体的核心代码实现存放在 `./src/phases/` 目录中。
